# F-0011: Pagos, Asignación FIFO y Crédito (por contrato)

## Objetivo

Controlar de forma precisa los pagos recibidos para un contrato, permitiendo:

- Conocer en todo momento el estado de cuentas por contrato.
- Aplicar pagos a devengos (`Accrual`) mediante regla FIFO dentro del contrato.
- Soportar sobrepagos y gestión de crédito aplicable a devengos futuros del mismo contrato.
- Responder, para una fecha concreta:
  - ¿Cuánto debería haber cobrado y de quién?
  - ¿Cuánto he cobrado realmente y de quién?
- Registrar en los recibos el detalle de asignaciones y crédito aplicado.

Los devengos cuyo `payer = OWNER` no generan registro de pagos en este modelo.

---

## Definiciones

- **Pago**: importe recibido asociado a un contrato.
- **Aplicación de pago**: asignación total o parcial de un pago a uno o varios devengos del contrato.
- **FIFO**: el pago se aplica primero al devengo pendiente más antiguo según un orden estable.
- **Crédito del contrato**: importe pagado por encima de la deuda pendiente del contrato, aplicable a futuros devengos del mismo contrato.
- **Saldo pendiente (por devengo y fecha T)**: `outstanding_debt_amount(T)` definido en Feature 8 (calculado con importes efectivos y considerando reversals).

---

## Entidades

### Payment

**Campos obligatorios**
- `contract_id`
- `payment_date`
- `amount`: decimal positivo
- `idempotency_key`: string

**Campos opcionales**

- `payer_tenant_id`: identificador del inquilino que realizó el pago (solo informativo)
- `reference`: string
- `notes`: string
- `external_id`: string
- `source`: enum { `MANUAL`, `AUTOGENERATED` }
- `reversal_of_payment_id`: opaque id

**Campos derivados (no persistidos)**
- `unallocated_amount = amount - sum(applied_amount)`
- `status`:
  - `UNALLOCATED` si `unallocated_amount = amount`
  - `ALLOCATED` si `unallocated_amount = 0`
  - `PARTIALLY_ALLOCATED` si `0 < unallocated_amount < amount`
- `effect_sign = -1 si reversal_of_payment_id existe, si no +1`
- `effective_amount = effect_sign * amount`
- `unallocated_effective_amount = effective_amount - sum(effective_applied_amount)`

---

### PaymentApplication

**Campos obligatorios**
- `payment_id`
- `accrual_id`
- `applied_amount`: decimal positivo
- `idempotency_key`: string

**Restricciones**
- `accrual.contract_id` debe ser igual a `payment.contract_id`.
- `applied_amount` ≤ saldo pendiente del devengo en el momento de la aplicación.
- `sum(applied_amount)` por pago ≤ `payment.amount`.
- Solo se puede aplicar a devengos con `payer = TENANT`.
- `PaymentApplication` es inmutable (append-only). Las correcciones no se hacen editando aplicaciones; se hacen revirtiendo el Payment completo y recreándolo.

**Campos derivados (no persistidos)**
- `effective_applied_amount = effect_sign(payment) * applied_amount`
---

## Alcance

- Modelo explícito `Payment` + `PaymentApplication`.
- FIFO determinista dentro del contrato con orden estable:
  1. `due_date` ascendente
  2. `accrual_date` ascendente
  3. `sequence` ascendente (entero incremental por contrato)
- Soporte de liquidaciones parciales.
- Soporte de sobrepago con crédito del contrato (sin entidad separada).
- Cálculo determinista del estado de cuenta por contrato y fecha.
- FIFO solo considera devengos con `outstanding_debt_amount(T) > 0` en el momento de aplicar.

---

## Capacidades

- Registrar pago de un contrato.
- Aplicar pago automáticamente por FIFO.
- Consultar por contrato:
  - deuda/saldo pendiente
  - crédito disponible
  - pagos por rango de fechas
  - aplicaciones por pago y por devengo
- Generar detalle para recibos:
  - qué devengos se liquidan
  - cuánto se aplica a cada devengo
  - cuánto crédito sobrante se genera y cómo se usa en pagos posteriores

---

Los listados y consultas de coleccion deben usar **paginacion obligatoria**.

Los parámetros de paginación configurables deben resolverse exclusivamente a través del configuration system definido en **EN-0202**.

No deben definirse mediante constantes hardcoded en adapters, servicios de aplicación o repositorios. Debe existir una única fuente de verdad para estos valores siguiendo la precedencia global de configuración:

`environment variables > config file > defaults`

## Fuera de alcance

- Conciliación bancaria automática.
- Reembolsos y devoluciones.
- Pagos del propietario a proveedores.
- Contabilidad completa de tesorería.
- Reversión parcial de pagos o de aplicaciones de pago (solo se permite reversión TOTAL del Payment).

---

## Reglas de negocio

1. Solo se registran pagos para cubrir devengos con `payer = TENANT`.
2. Todo pago pertenece a un contrato (`contract_id` obligatorio).
3. Un pago solo puede aplicarse a devengos del mismo contrato.
4. Asignación FIFO:
   - Se aplica únicamente a devengos con:
       - `payer = TENANT`
       - mismo `contract_id`
       - saldo pendiente > 0
   - Orden estable:
       1. `due_date` ascendente
       2. `accrual_date` ascendente
       3. `sequence` ascendente
5. Se permite aplicación parcial de un pago a un devengo.
6. Si `payment.amount` excede la suma de deuda pendiente del contrato:
   - el excedente queda como crédito del contrato (`unallocated_amount` agregado).
7. El crédito:
   - se aplica automáticamente a los siguientes devengos abiertos del contrato siguiendo FIFO.
   - debe reflejarse en recibos indicando qué parte del pago corresponde a crédito aplicado.
8. Para una fecha T, el sistema debe poder calcular por contrato:
   - Devengado exigible hasta T:
     Σ `effective_net_payable_amount` de Accruals del contrato con:
       - `payer = TENANT`
       - `due_date <= T` (si `due_date` es null, usar `accrual_date`)
       - considerando compensaciones vía `reversal_of_accrual_id`
   - Cobrado realmente hasta T:
     Σ `effective_amount` de Payments del contrato con `payment_date <= T`
     (incluye reversals totales con signo efectivo).
  - Balance económico del contrato hasta T (fuente de verdad):
    `balance_amount(T) = devengado_exigible_hasta_T - cobrado_realmente_hasta_T`
  - Saldo pendiente (contrato) hasta T:
    `debt_amount(T) = max(0, balance_amount(T))`
  - Crédito disponible (contrato) hasta T:
    `credit_amount(T) = max(0, -balance_amount(T))`
  - Total aplicado hasta T (trazabilidad/recibos):
    Σ `effect_sign(payment) * applied_amount` para todas las PaymentApplications
    del contrato con `payment_date <= T`.
  - Saldo pendiente hasta T:
       Σ `outstanding_debt_amount(T)` de Accruals exigibles del contrato.
8.A Invariante de conservación Payment↔Applications (por contrato y fecha T):
  - Para cualquier contrato y fecha T:
    `applied_total_amount(T) <= cobrado_realmente_hasta_T`
  - La diferencia `cobrado_realmente_hasta_T - applied_total_amount(T)` es exactamente el crédito no aplicado (siempre >= 0).
  - Este invariante depende del cumplimiento del punto 9.B.
9. No se cancelan pagos: se crea un pago compensatorio con reversal_of_payment_id.
9.A La reversión de un pago es SIEMPRE TOTAL:
    - Un Payment compensatorio (`reversal_of_payment_id`) debe tener el mismo `amount` que el original.
    - El efecto económico se deriva por signo (`effective_amount`), no por importes negativos persistidos.
9.B Reversión TOTAL de Payment y espejo de aplicaciones (invariante de conservación):
  - No se permite revertir/modificar una `PaymentApplication` de forma aislada.
  - Si existe un Payment reverso `R` con `reversal_of_payment_id = P.id`,
    entonces R DEBE replicar exactamente las aplicaciones de P:
  
      - Para cada `PaymentApplication (payment_id=P, accrual_id=A, applied_amount=x)`
        DEBE existir una `PaymentApplication (payment_id=R, accrual_id=A, applied_amount=x)`.
  
      - No se permiten aplicaciones adicionales en R que no existan en P.
  - Consecuencia matemática:
    Para cualquier devengo A y fecha T,
    el efecto neto de P y su reverso sobre `applied_effective_amount(T)` es 0.
  - Para corregir una asignación (FIFO incorrecto, error de imputación, etc.) se debe:
      1) crear el Payment reverso (reversal total) con sus aplicaciones espejo,
      2) crear un nuevo Payment correcto,
      3) generar sus aplicaciones nuevamente por FIFO.
10. Las aplicaciones deben ser trazables para reflejar en recibos:
  - asignación por devengo
  - crédito generado y aplicado posteriormente.
11. Las aplicaciones de pago deben registrarse explícitamente (PaymentApplication).
12. El sistema no recalcula automáticamente aplicaciones históricas salvo que se reviertan explícitamente.
13. Los cálculos por fecha deben considerar exclusivamente pagos y devengos con fecha <= T, aplicando importes efectivos (`effective_*`) para reflejar compensaciones (`reversal_of_*`).
14. FIFO se aplica sobre devengos con saldo pendiente calculado con importes efectivos (considerando reversals).
15. Crédito del contrato se calcula con importes efectivos; nunca se persiste como entidad aparte.
